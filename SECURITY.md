# 安全策略

---

## 报告安全问题

> 不要在公开 Issue、Discussion、PR 或日志中披露漏洞利用细节、密钥、Cookie、个人求职数据或浏览器会话。

### 报告渠道

```
  1. GitHub Private Vulnerability Reporting（优先）
     仓库主页 -> Security -> Report a vulnerability

  2. 联系仓库所有者（备用）
     如果上述入口不可用，请联系仓库所有者，仅提供建立
     安全沟通渠道所需的最少信息
```

---

## 范围

### 在范围内

| 类别 | 说明 |
|------|------|
| 数据泄露 | 未授权访问用户数据或证据存储 |
| 认证绕过 | 权限提升、Token 伪造、会话劫持 |
| 注入攻击 | SQL 注入、影响系统行为的 Prompt 注入 |
| SSRF / 文件读取 | 服务端请求伪造、任意文件访问 |
| 信息泄露 | 密钥通过日志泄露、CORS 配置不当、错误信息暴露 |

### 不在范围内

- 理论风险（无实际攻击路径）
- 第三方依赖的已知 CVE（通过 `uv audit` 定期管理）
- 基础设施配置问题（Docker、CI 等）
- 对项目维护者的社会工程攻击

## 安全实践

```
  代码提交              不提交 .env、密钥、Token、Cookie、会话或真实个人数据

  日志记录              不记录简历正文、面试回答、Token、Cookie、
                        签名 URL 或完整 Prompt

  输入处理              所有外部输入（网页、文档、模型输出、回调）
                        视为不可信

  外部写                默认关闭，需审批 + 幂等 + 审计

  URL 获取              限制协议、域名、重定向深度、响应大小和超时
                        启用 SSRF 防护

  依赖管理              锁定文件（uv.lock）+ uv audit
                        新增依赖需通过 Architecture Issue
```

---

## 禁止行为

未经明确授权，不得：

- 访问第三方账户或真实个人数据
- 执行会产生外部写入的测试（邮件、API 调用等）
- 公开发布未修复的安全问题细节
- 对生产系统执行暴力破解、DoS 或其他破坏性测试
